## 2024-05-10 - Parallelize networksetup sub-processes
**Learning:** In a single-threaded execution model, running synchronous bash commands (like `networksetup` via `subprocess.run()`) for each individual network interface sequentially can lead to significant delays and block the core thread. While parallelizing *reads* (`networksetup -getdnsservers`) is safe, parallelizing *writes* (`networksetup -setdnsservers`) is highly risky on macOS. Modifying the network state concurrently often leads to `configd` lock contention (e.g., throwing "The file has been changed by another application"), which can cause DNS changes to fail silently.
**Action:** By utilizing `concurrent.futures.ThreadPoolExecutor`, we parallelize the READ operations (`_get_current_dns_servers` and `_verify_dns_redirect`), leading to much faster response times while avoiding system race conditions during network enforcement writes.

## 2026-05-09 - Stat pre-check for watchdog hosts integrity
**Bottleneck:** The watchdog's `/etc/hosts` tamper detection ran a full `read_text()` + `SHA-256` hash every 250ms during active blacklist sessions. For a typical 10-50KB hosts file, that's ~200μs of I/O + CPU per tick × 4 ticks/sec = ~800μs/sec of completely wasted work when the file hasn't been touched.
**Fix:** Two-tier check — `stat()` returns `(mtime, size)` in ~2μs. If both match the cached values from the last `_enforce_block()`, skip the expensive read+hash entirely. Only fall through to the full SHA-256 when the filesystem metadata actually changes. This eliminates ~99% of unnecessary disk I/O on the hottest loop in the daemon.
**Safety:** If stat drifts (e.g. `touch` without content change), the hash still matches and we just update the cached stat. True tampering (content change) always triggers re-enforcement. The `_hosts_stat` cache is cleared on session cleanup alongside `hosts_hash`.
