## 2024-05-10 - Parallelize networksetup sub-processes
**Learning:** In a single-threaded execution model, running synchronous bash commands (like `networksetup` via `subprocess.run()`) for each individual network interface sequentially can lead to significant delays and block the core thread. While parallelizing *reads* (`networksetup -getdnsservers`) is safe, parallelizing *writes* (`networksetup -setdnsservers`) is highly risky on macOS. Modifying the network state concurrently often leads to `configd` lock contention (e.g., throwing "The file has been changed by another application"), which can cause DNS changes to fail silently.
**Action:** By utilizing `concurrent.futures.ThreadPoolExecutor`, we parallelize the READ operations (`_get_current_dns_servers` and `_verify_dns_redirect`), leading to much faster response times while avoiding system race conditions during network enforcement writes.
